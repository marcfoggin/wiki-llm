# AI Agent Protocol for NavTree Vault Maintenance

This document defines how AI agents that maintain the Obsidian vault
coordinate with the NavTree build pipeline. It is intended to be
referenced from agent system prompts.

## Overview

The vault content feeds a build pipeline (`build_navtree.py`) that
produces a navigation tree for web apps. Some notes appear at multiple
positions in that tree. Structural changes to those notes affect all
positions and must be logged so the build pipeline can validate them.

```
Obsidian vault ──→ .navtree/build_navtree.py ──→ .navtree/output/
(agents edit)       (CI runs)                     ├── redirects.toml  ← agents read
                                                  ├── 01/index.toml
                                                  ├── 01/content.md
                                                  └── ...

vault-changelog.toml  ← agents write this (in vault root: wiki/)
```

## Files the agent reads

### `output/redirects.toml`

Generated by the build script. Lists notes that appear at multiple
positions in the navigation tree. Fetch this before making structural
changes.

Read locally: `../.navtree/output/redirects.toml` (relative to vault root `wiki/`)

After CI: `https://raw.githubusercontent.com/{owner}/{repo}/main/wiki-llm/.navtree/output/redirects.toml`

```toml
[[multi_position]]
content = "Creation Care / Stewardship"
content_id = "creation-care"
positions = ["01/05/01", "02"]
primary = "01/05/01"
```

- `content` — the note title
- `content_id` — the slugified filename
- `positions` — all tree positions this note occupies
- `primary` — the position that holds the actual content; others are redirects

## Files the agent writes

### `vault-changelog.toml`

Written by the agent in the vault root whenever it makes structural
changes to a note listed in `redirects.toml`. The build script validates
these entries and archives them after a successful build.

```toml
[[change]]
date = "2026-06-02"
agent = "wiki-curator"
type = "rename"
note = "Creation Care"
new_name = "Creation Stewardship"
reason = "Aligning with updated terminology from source documents"
affected_positions = ["01/05/01", "02"]
```

**Fields:**

| Field | Required | Description |
|-------|----------|-------------|
| `date` | Yes | ISO date (YYYY-MM-DD) |
| `agent` | Yes | Agent identifier |
| `type` | Yes | One of: `rename`, `restructure`, `split`, `delete` |
| `note` | Yes | Original note filename (without .md) |
| `new_name` | For renames | New note filename |
| `action` | For restructure/split | What was changed and how |
| `reason` | Yes | Why this change was made |
| `affected_positions` | Yes | Copied from `redirects.toml` positions array |

## Rules by change type

### Content edits — free, no logging

Editing the content of any note (multi-position or not) is always safe.
Changes propagate to all tree positions automatically because content is
shared. No changelog entry needed.

### Adding new notes and links — free, no logging

Creating new notes and adding `[[wikilinks]]` to or from notes that are
NOT in `redirects.toml` requires no coordination.

### Renaming a multi-position note

1. Check `redirects.toml` — is the note listed?
2. If yes: rename the note, then log with `type = "rename"`
3. Include `new_name` and copy `affected_positions` from `redirects.toml`

### Restructuring links to a multi-position note

Adding or removing `[[wikilinks]]` that change where a multi-position
note appears in the tree.

1. Check `redirects.toml`
2. If the note is listed: make the change, then log with `type = "restructure"`
3. Describe what changed in `action`

### Splitting a multi-position note

Breaking one note into multiple notes.

1. Check `redirects.toml`
2. If listed: split the note, then log with `type = "split"`
3. Describe the split in `action`

### Deleting a multi-position note

1. Check `redirects.toml`
2. If listed: verify no other notes still link to it (or update those links)
3. Delete the note, then log with `type = "delete"`

### Notes NOT in `redirects.toml`

Can be renamed, restructured, split, or deleted freely without logging.

## System prompt snippet

Copy this into the agent's system prompt:

```
## NavTree vault coordination

Some notes in this vault appear at multiple positions in the navigation
tree built by build_navtree.py. Before making structural changes (rename,
restructure links, split, delete), fetch redirects.toml from the build
output to check if the note is multi-position:

  Local path: ../.navtree/output/redirects.toml  (relative to wiki/)
  URL: https://raw.githubusercontent.com/{owner}/{repo}/main/wiki-llm/.navtree/output/redirects.toml

Rules:
- Editing content: always safe, no logging needed.
- Adding new notes/links to non-multi-position notes: safe, no logging.
- Rename/restructure/split/delete a note listed in redirects.toml:
  make the change, then append an entry to vault-changelog.toml:

  [[change]]
  date = "YYYY-MM-DD"
  agent = "your-agent-name"
  type = "rename"              # rename | restructure | split | delete
  note = "Original Note Name"
  new_name = "New Name"        # renames only
  action = "What changed"      # restructure/split only
  reason = "Why"
  affected_positions = ["01/05/01", "02"]   # from redirects.toml

- Notes NOT in redirects.toml can be changed freely without logging.
```

## Build script behavior

When the build runs:

1. Reads `vault-changelog.toml` if it exists
2. Validates entries:
   - Warns if a renamed note's `new_name` cannot be found
   - Warns if a deleted note still exists
   - Warns if a restructured/split note cannot be found
3. Builds the tree from current vault state
4. Writes output (folder hierarchy, `search-positions.json`, `redirects.toml`)
5. Archives processed changelog entries to `vault-changelog.archive.toml`
6. Deletes `vault-changelog.toml`

The archive provides a history of structural changes for debugging.
