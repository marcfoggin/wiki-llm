# .navtree — NavTree Build Pipeline

Builds a NavTree v2 navigation structure from the Obsidian vault (parent
directory). The output is a hierarchy of numbered folders with TOML metadata
and Markdown content, designed to be consumed by web apps like bibel-wiki
and navtree-app.

This folder is dot-prefixed so Obsidian ignores it in its file explorer,
search, and graph view.

## Quick start

```bash
cd .navtree
python3 build_navtree.py

# Output is written to .navtree/output/
ls output/
```

## What it produces

```
.navtree/output/
├── config.toml              # Global settings (media CDN, languages)
├── search-positions.json    # content_id → tree positions (for search)
├── redirects.toml           # Multi-position content log (for AI agents)
├── 01/                      # Root node
│   ├── index.toml           # Metadata: content_id, title, extra fields
│   ├── content.md           # Markdown content (syntax preserved as-is)
│   ├── 01/                  # Child node
│   │   ├── index.toml
│   │   └── content.md
│   └── 02/
│       ├── index.toml
│       └── content.md
└── 02/                      # Redirect node (content lives at 01/...)
    └── index.toml           # Contains: redirect_to = "01/03/02"
```

The folder hierarchy IS the navigation tree. No separate `tree.json` needed.
Apps walk the numbered folders and read `index.toml` at each level, the same
convention bibel-wiki uses.

## Deduplication via redirects

When the same Obsidian note is linked from multiple places in the tree, it
gets content at its first occurrence and `redirect_to` at subsequent positions:

```toml
# output/02/index.toml — redirect, no content.md here
content_id = "creation-care"
title = "Creation Care / Stewardship"
redirect_to = "01/05/01"
```

The app follows the redirect to fetch content from the primary location.
`search-positions.json` tracks all positions so search results can link to any.

## AI agent integration

The Obsidian vault is maintained by AI assist agents that can make both content
edits and structural changes. Two files coordinate between the agents and the
build pipeline:

### `output/redirects.toml` — build tells agents what's multi-position

Generated by the build script. Agents read this before making structural
changes to check if a note is multi-position.

### `vault-changelog.toml` — agents tell the build what changed

Written by AI agents in the vault root (parent of `.navtree/`) whenever they
make structural changes to multi-position content. The build script validates
entries and archives them after a successful build.

The coordination rules are defined in `CLAUDE.md` (in the vault root)
under the "NavTree vault coordination" section.

## Configuration

Edit `build.toml` to configure the build:

```toml
vault_path = ".."            # Parent directory is the vault
output_dir = "output"        # Output within .navtree/

skip_prefixes = ["Sources/", "source"]   # Link prefixes to skip
exclude_roots = ["log"]                  # Notes to exclude from roots
```

Root notes are discovered from `../index.md` — each `[[wikilink]]` under
a `##` heading becomes a root in the tree.

## CI pipeline

The build runs automatically when vault markdown files change:

```
push to wiki-llm/wiki/**/*.md or wiki-llm/.navtree/**
        │
        └─ PlateauPerspectives/.github  (org-level workflow repo)
               workflows/build.yml
               │
               ├─ cd wiki-llm/.navtree
               ├─ python build_navtree.py
               ├─ commit output/ (including redirects.toml)
               └─ archive vault-changelog.toml entries
```

The workflow lives in the org-level
[PlateauPerspectives/.github](https://github.com/PlateauPerspectives/.github)
repo, not inside this repository.
